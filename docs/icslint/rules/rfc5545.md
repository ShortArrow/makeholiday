# `RFC5545/*` — cardinality and required fields

Drawn directly from the RFC 5545 §3.6.1 cardinality table. These
rules catch the violations that break at least one real-world
consumer in the wild — missing required properties, duplicate
single-occurrence properties, conflicting time anchors, and broken
date ordering.

All eight rules ship in `icslint` v0.2.0.

---

## `RFC5545/required-uid` — error

`VEVENT` missing `UID`. RFC 5545 §3.6.1 requires `UID` on every
`VEVENT`; without it, consumers cannot deduplicate, cross-reference,
or recognize update events.

**Trigger**

```ics
BEGIN:VEVENT
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260430
SUMMARY:Showa Day
END:VEVENT
```

**Fix**: add a `UID:<globally-unique-string>` line.

---

## `RFC5545/required-dtstamp` — error

`VEVENT` missing `DTSTAMP`. RFC 5545 §3.6.1 requires `DTSTAMP` on
every `VEVENT`; clients use it to order revisions of the same event.

**Trigger**

```ics
BEGIN:VEVENT
UID:event-1
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260430
END:VEVENT
```

**Fix**: add `DTSTAMP:<UTC ISO-8601 basic-format>`, e.g.
`DTSTAMP:20260101T000000Z`.

---

## `RFC5545/required-dtstart` — error

`VEVENT` missing `DTSTART`. RFC 5545 §3.6.1 requires `DTSTART` when
the enclosing `VCALENDAR` does not carry a `METHOD` property (the
common case). Without `DTSTART`, the event has no anchor on a
timeline.

**Trigger**

```ics
BEGIN:VEVENT
UID:event-1
DTSTAMP:20260101T000000Z
DTEND;VALUE=DATE:20260430
SUMMARY:Showa Day
END:VEVENT
```

**Fix**: add `DTSTART;VALUE=DATE:<YYYYMMDD>` for date events or
`DTSTART:<YYYYMMDDTHHMMSSZ>` for timed events.

---

## `RFC5545/duplicate-summary` — error

`SUMMARY` appears more than once inside a `VEVENT`. RFC 5545 §3.6.1
permits at most one. Multiple occurrences usually indicate a merge
artifact; consumers either pick the last value silently or reject
the event entirely.

**Trigger**

```ics
BEGIN:VEVENT
UID:e1
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260430
SUMMARY:original
SUMMARY:overwritten
END:VEVENT
```

**Fix**: keep one `SUMMARY`, drop the rest.

---

## `RFC5545/duplicate-dtstart` — error

`DTSTART` appears more than once inside a `VEVENT`. RFC 5545 §3.6.1
permits at most one. A duplicate either silently overrides the first
or breaks consumers expecting a single anchor.

**Trigger**

```ics
BEGIN:VEVENT
UID:e1
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTSTART;VALUE=DATE:20260501
DTEND;VALUE=DATE:20260430
SUMMARY:s
END:VEVENT
```

**Fix**: keep one `DTSTART`.

---

## `RFC5545/conflicting-end-and-duration` — error

Both `DTEND` and `DURATION` are present on the same `VEVENT`. RFC
5545 §3.6.1 lists them as mutually exclusive. When both are set the
effective end is ambiguous and consumers disagree on precedence.

**Trigger**

```ics
BEGIN:VEVENT
UID:e1
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260430
DURATION:P1D
SUMMARY:s
END:VEVENT
```

**Fix**: keep either `DTEND` or `DURATION`, never both.

---

## `RFC5545/end-before-start` — error

`DTEND` is not strictly later than `DTSTART`. For DATE-valued events
`DTEND` is the non-inclusive end, so a single-day event runs from
`DTSTART=20260429` to `DTEND=20260430`. `DTEND <= DTSTART` produces
a zero- or negative-duration event that most clients silently drop.

**Trigger**

```ics
BEGIN:VEVENT
UID:e1
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260429
SUMMARY:s
END:VEVENT
```

**Fix**: set `DTEND` to a date strictly later than `DTSTART`. For a
single-day event use `DTSTART+1`.

---

## `RFC5545/empty-summary` — warning

`SUMMARY` is present but its value is the empty string (or
whitespace only). `SUMMARY` itself is optional, but an empty value
is almost always a merge artifact or a forgotten template
placeholder; most clients render the event with a blank title.

**Trigger**

```ics
BEGIN:VEVENT
UID:e1
DTSTAMP:20260101T000000Z
DTSTART;VALUE=DATE:20260429
DTEND;VALUE=DATE:20260430
SUMMARY:
END:VEVENT
```

**Fix**: either drop the `SUMMARY` line, or fill in a real title.
